# Generated SegmentStream Runtime

This directory is generated by `segmentstream init` and `segmentstream run`.

Do not edit files here directly. The CLI may delete and recreate this directory
at any time. Update `../segmentstream.yml` and run `segmentstream run` instead.

Use `segmentstream run` from the project root to regenerate this directory,
restart the local Dagster/dbt runtime, and run materialization.
The first run can take a few minutes while Docker downloads and builds the
local environment; later runs should be faster.

Materialization is partitioned by day. `segmentstream run` executes the last 30
UTC daily partitions by default.

## Sources And Identity

Project-owned source packages live outside this generated directory under
`../sources/`. Event sources implement the `events` contract; conversion
sources implement the `conversions` contract; identity key sources implement the
`identity_keys` contract and emit timestamped observation rows with `date`,
`observed_at`, `anonymous_id`, `key_name`, and `key_value`. Analytics-core
compresses those observations into daily key spans before building links.

Declare source packages and identity link keys in `../segmentstream.yml`:

```yaml
sources:
  - name: ga4
    path: ./sources/ga4
  - name: crm_conversions
    path: ./sources/crm_conversions
  - name: sdk_identity
    path: ./sources/sdk_identity

identity:
  keys:
    - name: user_id
      tier: deterministic
      window_days: 180
      max_distinct_anonymous_ids: 1000
      scope: project
    - name: ip_address
      tier: probabilistic
      window_days: 3
      max_distinct_anonymous_ids: 100
      scope: source
```

Scaffold and verify source packages from the project root:

```sh
segmentstream source scaffold ga4 --type events
segmentstream source scaffold crm_conversions --type conversions
segmentstream source scaffold sdk_identity --type identity_keys
segmentstream source verify ga4
segmentstream source verify crm_conversions
segmentstream source verify sdk_identity
```

## Files

- `docker-compose.yml` starts the local SegmentStream runtime.
- `Dockerfile` defines the local Python/dbt/Dagster image.
- `dagster.yaml` configures Dagster's internal run and event storage in the local Postgres service.
- `dbt_project.yml` and `profiles.yml` are generated from the embedded SegmentStream runtime template.
- `dagster/` reads `../segmentstream.yml`, generates dbt composition files, and runs materialization.
- `packages.yml` is generated by Dagster from analytics-core and declared `sources`.
- Core dbt models, including `events`, `conversions`, `identity_keys`, and `identity_links`, are installed from the analytics-core dbt package.
- `.env` is generated from `segmentstream.yml` and is used by Docker Compose and dbt.
- `dbt/`, `target/`, and `logs/` are runtime working directories.

Dagster run history, event logs, and asset state are stored in the
`segmentstream-postgres-data` Docker volume. Regenerating `.segmentstream/` does
not delete that database volume.

## Debugging

The manual Docker debugging equivalent after generation is:

```sh
cd .segmentstream
docker compose up -d --build --force-recreate
```

Then open http://localhost:3000 to inspect project assets, runs, and backfills.
